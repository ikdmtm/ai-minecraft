AI Minecraft 配信プロジェクト
仕様整理ドキュメント
対象キャラクター名: 星守レイ
用途: 企画・実装・配信運用の共通土台
この文書は、ここまでの会話で合意した内容を今後の設計・実装・運用で再利用できるように、漏れがない形で整理したものである。未確定事項は明示し、それ以外は現時点の暫定決定として扱う。

# 1. プロジェクトの目的と前提
本プロジェクトの目的は、AI に Minecraft をプレイさせ、その様子を 24 時間自動でライブ配信し、配信収益および関連アーカイブ資産を生み出すことである。単に AI がゲームを動かせること自体は差別化になりにくいため、「見続ける理由」が成立する番組設計を優先する。
- 主目的は配信収益であり、技術デモだけで終わらせない。
- 視聴者にとっての価値は「何をしているか分かること」「次が気になること」「キャラに愛着が湧くこと」に置く。
- AI の完全自律性よりも、配信としての見やすさ・面白さ・継続性を上位に置く。
# 2. 配信コンセプト
基本コンセプトは「AI VTuber 星守レイが Minecraft ハードコアでどれだけ長く生き残れるかを観測する長期生存実験」である。
- 最優先目標: ハードコアでできるだけ長く生き残ること。
- 主な見どころ: 常に死亡リスクがある中での判断、撤退、拠点づくり、記録更新の緊張感。
- シリーズ性: 死亡したらその個体の配信は終了し、新しい配信枠で次の試行を開始する。
- 視聴体験: AI の思考・判断を実況として出し、状況理解しやすい配信にする。
重要な考え方: 「AI が上手い」よりも、「賢く判断しながらも死ぬかもしれない緊張感」が配信価値の中心になる。AI を強くしすぎてドラマが消える状態は避ける。
# 3. 番組として成立させるための柱
1. 大目標を一つに絞る。現時点の大目標は「生存記録を伸ばすこと」。
2. その時々の中目標を持たせる。例: ベッド確保、鉄装備、食料安定化、拠点強化、ネザー到達。
3. 安全時のみサブタスクを自律生成させる。例: 木材不足を補う、倉庫整理、周辺探索など。
4. 思考を配信用に見える形で出す。長いテンプレ台詞ではなく、その場の判断や考えをそのまま表出する。
# 4. 画面UI仕様
UI は「ゲーム画面だけでは分からない情報」に絞り、文字が多くなりすぎない構成にする。HP・食料などゲーム内ですでに表示される情報は、配信UIでは再掲しない。
### 除外したもの
- HP、食料ゲージ: ゲーム内表示に任せる。
- 最高記録との差分表示: 情報過多を避けるため現時点では採用しない。
- 感情ゲージなどの補助UI: 本質ではないため現時点では不要。
# 5. 音声実況方針
音声付き実況は採用する。理由は、長時間配信では視聴者が常に画面を見ているわけではなく、音声があることで状況を追いやすくなるためである。
- 常に喋り続けるのではなく、判断や危機時を中心に話す。
- 過剰なテンションや絶叫ではなく、聞き疲れしない落ち着いた話し方を基本とする。
- セリフテンプレート固定ではなく、その時の思考・判断をそのまま出す方針にする。
- 結果として、実況者というより「AI の心の声」が聞こえる配信を目指す。
# 6. キャラクター設定: 星守レイ
キャラクターは「AI らしさ」と「VTuber としての愛着」の両立を狙う。名前は硬すぎる機能名ではなく、視聴者が呼びやすく、推しやすいものとして「星守レイ」を採用する。
### キャラ運用上の注意
- テンプレセリフ集で人格を作るのではなく、思考の一貫性で人格を出す。
- 完全無機質にはせず、危険時や拠点喪失の可能性がある時に少し情緒が出る余地を残す。
- 大声で騒ぐタイプにはしない。ハードコアの緊張感と長時間配信の視認性を優先する。
- 死亡しても人格はリセットしない。世界線 / 個体は更新されても「星守レイ」は同一キャラクターとして継続する。
# 7. ゲーム内行動原則
1. 生存を最優先する。
2. 不確実な戦闘を避ける。
3. 安全が確保されていない状態では無理な探索をしない。
4. 食料・寝床・基本装備の安定化を先に行う。
5. 安全時のみサブタスクを自律生成する。
6. 死亡原因となるパターンを記録し、次回以降の判断に反映する。

ただし、生存優先が「何も起きない退屈な配信」を生む場合、それ自体が配信としての失敗になる。安定期における行動判断の詳細はセクション 8（自律ペーシング設計）で定義する。

# 8. 自律ペーシング設計

長時間配信では「安全だが何も起きない時間」が最大の離脱要因になる。一方で、外部システムが行動を強制すると AI の自律性が失われ、不自然な挙動につながる。本設計では、AI 自身がペーシングを判断できる仕組みを最小限の構成で実現する。

### 設計方針
- ペーシングの判断主体は AI 自身とする。外部システムが「次は探索しろ」のように行動を指示することはしない。
- 外部システムの役割は、AI が自力では把握しにくい「時間経過」や「行動の偏り」を事実情報として提供することに限定する。
- AI のプロンプトに、生存と配信価値のバランスを取るための判断基準を組み込む。

### 外部システムが AI に渡す状態情報
以下の情報を、AI への入力コンテキストに毎ターン含める。数値の提示のみで、解釈や指示は付けない。

- **現在の行動カテゴリ**: 採掘 / 建築 / 探索 / 戦闘 / 待機 / 移動 / クラフト / 農業 など
- **同一カテゴリの継続時間**: 現在の行動を何分続けているか
- **最後に行動カテゴリが変わってからの経過時間**: 同一行動の繰り返し検知用
- **全体生存時間**: 今回の試行の経過時間
- **現在の進捗フェーズ**: 初期（装備なし）/ 安定期（拠点・鉄装備あり）/ 発展期（ダイヤ・エンチャント）/ 挑戦期（ネザー以降）

行動カテゴリの分類は、bot の直近の行動ログから機械的に判定する。厳密な分類精度は不要で、大まかな傾向が伝われば十分である。

### AI に組み込む判断基準（プロンプト設計指針）
AI のシステムプロンプトに以下の判断基準を含める。これは「ルール」ではなく「価値観」として記述し、状況に応じた柔軟な判断を促す。

1. **危険が迫っている時**: 生存を最優先する。ペーシングは考慮しない。
2. **生存が安定している時**: 以下を自分で判断する。
   - 同じ種類の行動を長時間続けていないか
   - 視聴者から見て「次に何が起きるか」が期待できる状態か
   - 現在の装備・準備であれば、少しリスクのある挑戦に踏み出せるか
   - 今の進捗フェーズに対して、次の中目標に進むべきタイミングではないか
3. **核心となる価値観**: 「安全だが退屈な状態」は、配信としては「危険だが見どころがある状態」より悪い。生存リスクだけでなく、視聴者離脱リスクも判断に含めること。

### 段階的リスクテイクの考え方
AI は進捗フェーズに応じて、取るべきリスクの水準を自分で引き上げる。以下は目安であり、AI が状況を見て判断する。

- **初期**: リスク回避最優先。まず生き残る基盤を作る。
- **安定期**: 拠点周辺の探索、洞窟探検など中程度のリスクを取り始める。
- **発展期**: ネザー到達、要塞探索など明確な目標に向けて行動する。
- **長期安定**: 大型建築、トラップ作成、未踏地探索など、視覚的に変化のある活動を増やす。

### セーフティネット: 停滞検知
AI の自律判断を基本としつつ、極端な停滞に対して 1 段階だけ外部から介入する仕組みを設ける。

- **条件**: 同一行動カテゴリが連続 40 分を超えた場合（閾値は運用で調整）
- **介入方法**: AI への入力コンテキストに以下の追加情報を付与する。「⚠ 同じ行動が長時間続いています。視聴者の関心を維持するために、行動の変化を検討してください。」
- **性質**: これは命令ではなく追加情報である。最終的な行動決定は AI が行う。
- **段階**: 1 段階のみ。それ以上の強制介入は行わない。AI が判断した上でなお同じ行動を続ける場合は、それを尊重する（例: 大規模建築の途中など、合理的な理由がある場合）。

### この設計で意図的に行わないこと
- 外部タイマーによる強制的な行動切り替え
- 「N 分ごとに探索イベントを発生させる」といったスクリプト的ペーシング
- 行動の面白さを外部システムがスコアリングして制御すること
- 視聴者数の変動に連動した行動変更（MVP スコープ外）

# 9. 配信スタイルの決定事項
配信スタイルに関して、ここまでで次を採用する。
- 生存中は同じ配信枠を継続する。
- 死亡したらその配信は終了し、次は新しい配信枠で開始する。
- 初回から定型自己紹介のような手動感の強い演出は入れず、仕組みとして自然に回る設計にする。
- 配信の価値は、AI の実況、記録更新の緊張感、そして継続シリーズとしての物語性に置く。

### 運用モード
システムは 2 つの運用モードを持ち、設定で切り替える。

**MANUAL モード（MVP / テスト期）**
- オペレーターが配信開始トリガーを発行すると、1 回の試行（スポーン→プレイ→死亡）を実行する。
- 死亡後、配信を終了し IDLE 状態に戻る。次のトリガーがあるまで何もしない。
- インスタンスは試行しないときに停止できるため、コストを大幅に抑えられる。
- テスト時は YouTube 配信なしでローカル録画のみで実行することも可能。

**AUTO モード（本番運用期）**
- 死亡後、クールタイムを経て自動的に次の配信を開始する。24 時間自動運用。
- セクション 11 の YouTube 運用ポリシー（クールタイム、日次上限制御）が適用される。
- MANUAL モードで動作が安定し、視聴者がついてきた段階で移行する。

両モードでオーケストレーターのコア部分（状態機械、Bot 制御、LLM 連携、映像配信）は同一。違いは死亡後の遷移先（IDLE vs COOL_DOWN）のみ。
# 10. プラットフォーム方針とアーカイブ方針
配信プラットフォームは YouTube を主戦場とする。理由は、Twitch には 1 回の配信で 48 時間の上限があり、将来的に長寿個体がその上限を超える可能性を考慮すると制約が大きいためである。一方 YouTube は、現時点で主に問題になるのが長時間配信の自動アーカイブ制約であり、これについてはローカル録画を本体にすることで回避する。
- ライブ配信: YouTube
- バックアップ / 本体保存: ローカル録画を常時実施
- 長期資産化: ローカル録画を整理して後でアップロード
- 配信プラットフォーム上のアーカイブは補助であり、主保存先にはしない
運用上の重要原則: 「ライブは視聴導線、ローカル録画は資産」という分担で考える。配信サイトの自動アーカイブ仕様に依存しない。
# 11. YouTube運用ポリシー
YouTube ではライブ作成回数の日次制約に配慮し、死亡のたびに即座に無制限で新規配信を作るのではなく、クールタイムと再開ポリシーを設ける。
- 死亡したら配信終了後にクールタイムを入れてから次枠を作成する。
- 通常死亡時の再開待ちは 5〜15 分程度を基準とする。
- 短時間の連続死亡時は待機時間を段階的に延長する。
- 日次ライブ作成上限に近づいた場合は、より長い待機時間を取る。
- 上限到達時は自動で待機モードに入り、一定時間後または翌日まで再開を抑制する。
# 12. 自動運用の全体構成
自動運用の核心は「AI に全部やらせること」ではなく、配信全体を制御するオーケストレーターを作ることである。AI は判断と実況を担当し、それ以外の状態管理や復旧は通常コードで確実に制御する。この構成は MANUAL / AUTO 両モードで共通であり、モードの違いは死亡後の遷移先（IDLE or COOL_DOWN）のみで吸収する。
# 13. 配信制御の状態遷移
実装は状態機械で管理する。これにより、死亡、API失敗、再接続、再起動などの例外ケースでも挙動を明確にできる。

- **IDLE**: 待機中。オペレーターのトリガーまたは AUTO モードの自動遷移を待つ。MANUAL モードのデフォルト状態。
- **BOOTING**: 起動中。必要なプロセスと接続を初期化する。
- **PREPARING_STREAM**: 次の配信の準備を行う。
- **LIVE_RUNNING**: 通常配信中。
- **DEATH_DETECTED**: 死亡を検知した直後の処理状態。
- **ENDING_STREAM**: 現在の配信を終了する。
- **COOL_DOWN**: 次枠までの待機時間（AUTO モードのみ使用）。
- **CREATING_NEXT_STREAM**: 次の配信枠を作成する。
- **RECOVERING / RETRY_WAIT**: 障害や API 失敗からの復旧待ち。
- **SUSPENDED_UNTIL_NEXT_DAY**: 日次制限等により再開を抑制する待機状態（AUTO モードのみ）。

### モード別の遷移の違い

```
[MANUAL モード]
IDLE →(トリガー)→ BOOTING → PREPARING_STREAM → LIVE_RUNNING
  → DEATH_DETECTED → ENDING_STREAM → IDLE（停止。次のトリガーまで待機）

[AUTO モード]
BOOTING → PREPARING_STREAM → LIVE_RUNNING
  → DEATH_DETECTED → ENDING_STREAM → COOL_DOWN → CREATING_NEXT_STREAM
  → PREPARING_STREAM → LIVE_RUNNING → ...（繰り返し）
```

MANUAL モードのトリガーは CLI コマンド、HTTP API エンドポイント、またはスケジューラ（cron 等）で実行可能とする。
# 14. 画面合成とシーン管理（FFmpeg）
FFmpeg のオーバーレイ切り替えにより、以下のシーンを表現する。OBS は使用しない。
- **LIVE シーン**: 通常プレイ画面 + 最低限の情報UI（生存時間、目標、危険度、実況字幕）。
- **DEATH シーン**: 死亡演出。今回の生存時間、最高記録、死因を短時間表示。
- **BOOT / WAITING シーン**: 次の観測準備中または IDLE 中の待機画面。
- 常時ローカル録画を有効化し、長時間運用時のアーカイブ消失を避ける。
- 録画は巨大な単一ファイルではなく、時間単位で分割する方針が望ましい。
# 15. 死亡から次配信開始までの想定フロー

共通ステップ（両モード共通）:
1. ゲーム側または bot 側で死亡イベントを検知する。
2. FFmpeg のオーバーレイを DEATH 表示に切り替え、短い死亡演出を出す。
3. 現在の配信枠を終了する。
4. 録画ファイルを確定し、S3 転送をトリガーする。

MANUAL モードの場合:
5. IDLE 状態に遷移する。次のトリガーまで待機。
6. （オペレーター判断でインスタンスを停止してもよい）

AUTO モードの場合:
5. クールタイムへ移行する。
6. クールタイム経過後、次の配信枠を作成する。
7. 新しいワールド / 試行を準備する。
8. FFmpeg の送信先や表示情報を次世代向けに更新する。
9. 次の配信を開始し、LIVE_RUNNING へ戻る。
# 16. 監視・復旧設計
24 時間運用では、通常時の動作以上に「落ちた時に自動で戻れるか」が重要である。配信の継続性を守るため、監視用の親プロセスを用意し、各子プロセスの生存確認を行う。
オーケストレーターが以下のヘルスチェックを **10 秒間隔** で実行する。

| チェック対象 | 判定方法 | 異常判定基準 | 復旧アクション |
|---|---|---|---|
| Minecraft Server | localhost:25565 への TCP 接続確認 | 3 回連続失敗（30 秒） | systemctl restart minecraft-server |
| Minecraft Client | Xvfb の DISPLAY :99 にウィンドウが存在するか | 3 回連続失敗 | systemctl restart minecraft-client |
| Mineflayer Bot | 最後の行動ログのタイムスタンプ | 60 秒以上行動なし | bot プロセスを再起動 |
| FFmpeg (配信) | プロセス存在 + RTMP 送出バイト数が増加しているか | 30 秒間バイト数変化なし | systemctl restart ffmpeg-stream |
| FFmpeg (録画) | ローカル録画ファイルのサイズが増加しているか | 60 秒間サイズ変化なし | FFmpeg 再起動 |
| VOICEVOX | localhost:50021 への HTTP ヘルスチェック | 3 回連続失敗 | docker restart voicevox |
| YouTube 配信 | YouTube Live Streaming API でストリームステータス確認 | ステータスが inactive | RTMP 再接続 or 配信枠再作成 |
| LLM API | 直近の呼び出し成功率 | 5 回連続失敗 | リアクティブ層のみで継続。3 分後にリトライ |
| SQLite DB | 書き込みテスト | 書き込み失敗 | ログに警告。EBS の空き容量確認 |

実装上の原則: AI は判断と実況だけを担当し、死亡判定・配信枠・再試行・復旧は deterministic な通常コードで実装する。
# 17. データとして保持すべき情報
- 現在の世代番号 / 個体番号
- 現在の生存開始時刻
- 最高記録
- 直前の死亡理由や状況ログ
- 現在の配信 ID / ストリーム ID
- 当日作成したライブ配信数
- クールタイム開始時刻 / 次回再開予定時刻
- ワールド識別子、設定、初期シード等（必要に応じて）
- AI の行動ログおよび実況ログ
- 現在の行動カテゴリおよびその継続時間（ペーシング用）
- 現在の進捗フェーズ
- 過去の死亡履歴（世代番号、生存時間、死因、学んだ教訓の要約）
- 運用設定（配信タイトルテンプレート、概要欄テンプレート、クールタイム等）
# 18. このプロジェクトの優先順位
配信としての成功を考えると、開発の優先順位は次の通りとする。
1. 死亡しにくく、最低限まともに動くこと。
2. 視聴者が今何をしているか理解できること。
3. 危機や節目にドラマが生まれること。
4. 長時間の自動運用に耐えること。
5. 完全自律性を高めること。
# 19. 現時点で確定していること / 未確定のこと
### 確定していること
- キャラクター名は星守レイ。
- 配信コンセプトは AI VTuber による Minecraft ハードコア長期生存実験。
- 生存中は同じ配信枠、死亡したら新しい配信枠。
- UI は生存時間・最高記録・目標・危険度・実況テキストを中心に構成する。
- ライブ配信は YouTube を使用し、ローカル録画を必須とする。
- YouTube の日次ライブ作成上限にはクールタイムで対応する。
- ペーシングは AI の自律判断を基本とし、外部からは時間経過や行動カテゴリの事実情報のみ提供する。停滞検知（40 分超）による 1 段階の情報追加をセーフティネットとする。
- インフラは AWS g4dn.xlarge（東京リージョン）に全コンポーネントを集約する単一インスタンス構成。
- 映像は Minecraft Client + Xvfb + VirtualGL で GPU 描画し、FFmpeg + NVENC でエンコード・配信する。OBS は使用しない。
- Bot 制御は Mineflayer (Node.js)。LLM は API ベース（Claude / GPT-4o）。TTS は VOICEVOX (CPU, Docker)。
- オーケストレーターは Node.js (TypeScript) + 状態機械。プロセス管理は systemd。
- 録画は 1 時間分割、NVMe 一時保存 → S3 定期転送 → YouTube アップロード確認後に S3 からも削除。命名規則: `gen{世代番号}_{YYYYMMDD}_{HHmmss}.mp4`。
- 運用は MANUAL モード（手動トリガー）で開始し、安定後に AUTO モード（24 時間自動）へ移行する段階的アプローチを取る。
- Bot の行動判断は 2 層構造: リアクティブ層（ハードコード、即時応答）で生存を保証し、戦略層（LLM、可変間隔）で創造的判断と実況を行う。
- 記憶・学習の仕組み: 死亡時に教訓を LLM で生成し、SQLite に蓄積。毎ターンの LLM 入力に直近 5 件の死亡履歴を含める（セクション 22）。
- YouTube メタデータ: タイトル・概要欄・サムネイル・タグの自動生成テンプレートを定義済み（セクション 24）。
- 配信開始フロー: BOOTING → PREPARING_STREAM → LIVE_RUNNING の全ステップを定義済み（セクション 25）。
- ダッシュボード: Express.js HTTP サーバー + 1 ページ HTML。SSH トンネル経由。API 8 エンドポイント定義済み（セクション 26）。
- ヘルスチェック: 10 秒間隔で 9 項目を監視。具体的な異常判定基準と復旧アクションを定義済み（セクション 16）。
- オーケストレーター耐障害: systemd Restart=always + SQLite 状態復元。復旧フロー定義済み（セクション 27）。
- 障害シナリオ: 11 パターンの障害と自動復旧手順を定義済み（セクション 28）。
- Minecraft バージョン: Java Edition、Mineflayer 対応の最新安定版を固定使用（セクション 29）。

### ストレージ戦略

```
NVMe ローカル (125GB)
  ↓ cron (15分間隔) で転送
S3 Standard (一時保管)
  ↓ YouTube アップロード完了を確認
  ↓ 確認済み → S3 から削除
  ↓ 未アップ → 保持（手動 or 自動アップロード後に削除）
```

- 録画は 1 時間単位でファイル分割する（FFmpeg の `-segment_time 3600`）。
- ファイル命名規則: `gen{世代番号}_{YYYYMMDD}_{HHmmss}.mp4`
- S3 転送は cron + AWS CLI。転送完了後に NVMe のローカルファイルを削除。
- YouTube にアップロード完了が確認できた録画は S3 からも削除する。これにより S3 の常時保持量を最小化する（月 $2〜5 程度に抑制）。
- NVMe は一時領域であり、インスタンス停止で消失する。重要データ（SQLite DB、設定ファイル等）は EBS に配置する。
- MANUAL モードでインスタンス停止前に、NVMe 上の未転送録画がないことを確認するスクリプトを用意する。

| レイヤー | 手段 | 内容 |

# 21. 技術スタック

### 映像パイプライン

Minecraft の映像を配信するために、以下の構成を取る。

```
Minecraft Client (Java Edition)
  │  GPU 描画 (NVIDIA T4)
  │  Xvfb (仮想ディスプレイ :99, 1920x1080)
  │  VirtualGL (GPU アクセラレーション)
  ↓
FFmpeg (x11grab でキャプチャ)
  │  NVENC (T4 ハードウェアエンコード, H.264)
  │  + オーバーレイフィルタ (UI 合成)
  │  + PulseAudio 入力 (VOICEVOX 音声 + Minecraft ゲーム音)
  ↓
├→ YouTube RTMP (ライブ配信)
└→ ローカルファイル (録画保存 → S3)
```

- **Xvfb**: 仮想フレームバッファ。物理ディスプレイなしで X Window を提供する。
- **VirtualGL**: Xvfb 上のアプリケーションに GPU レンダリングを提供する。これがないと Minecraft が CPU ソフトウェアレンダリングになり、パフォーマンスが大幅に低下する。
- **NVENC**: NVIDIA T4 のハードウェアエンコーダ。CPU 負荷をほぼかけずに H.264 エンコードが可能。
- **OBS は使わない**: FFmpeg による直接配信の方が 24 時間自動運用で安定する。GUI アプリである OBS は headless 環境での自動復旧が困難。

### アバター表示

AI VTuber として視聴者の愛着を得るため、星守レイのアバターを配信画面上に常時表示する。

#### アバター方式: スプライトベース（MVP）→ Live2D（Phase 2）

**MVP（スプライトベース）:**
- 事前に複数の表情パターンを透過 PNG で用意する（立ち絵ベース）
- 配信画面の**右下**に 300x400px 程度で表示
- オーケストレーターが `threat_level` や行動状態に応じて表情を切り替える

| 表情 | トリガー条件 |
|---|---|
| 通常（にこやか） | threat_level: low、通常行動中 |
| 真剣 | threat_level: medium、戦闘・採掘中 |
| 焦り | threat_level: high、HP 低下・逃走中 |
| 恐怖 | threat_level: critical、即死危機 |
| 喜び | 記録更新、レアアイテム入手、進捗フェーズ移行 |
| 考え中 | LLM 応答待ち中 |

- **簡易リップシンク**: VOICEVOX の音声再生中は口を開けた差分画像に切り替える（200ms 間隔で口開き/口閉じを交互表示）。音声が再生されていないときは口閉じ。
- 必要な画像素材: 6 表情 × 2（口開き/口閉じ）= **12 枚の透過 PNG**
- 切り替え: オーケストレーターが現在の表情ファイルパスを JSON に書き出し、描画プロセスがオーバーレイに合成

**Phase 2（Live2D）:**
- Live2D Cubism SDK for Web を使い、より自然なアニメーション（瞬き、揺れ、口パク同期）を実現
- Chromium の Headless モードで Live2D を描画し、FFmpeg にフィードとして渡す
- VOICEVOX の音素情報と連動した精密な口パク
- Phase 2 への移行は MVP の配信が軌道に乗ってから判断

#### 配信画面レイアウト

```
┌──────────────────────────────────────────────────┐
│                                                  │
│                                                  │
│              Minecraft ゲーム画面                  │
│              (1920 x 1080)                        │
│                                                  │
│  ┌──────────┐                                    │
│  │ 生存時間  │                        ┌────────┐ │
│  │ 最高記録  │                        │        │ │
│  │ 目標     │                         │ アバター │ │
│  │ 危険度   │                         │ (300x  │ │
│  └──────────┘                        │  400)  │ │
│                                      └────────┘ │
│  ┌──────────────────────────────────────────────┐│
│  │ 💬 実況テキスト（字幕）                        ││
│  └──────────────────────────────────────────────┘│
└──────────────────────────────────────────────────┘
```

- 左上: ステータス情報（生存時間、最高記録、目標、危険度）
- 右下: アバター（ゲーム画面に若干被る形で表示。半透明の影を敷いて読みやすくする）
- 下部: 実況テキスト字幕（現在の commentary を表示。3〜5 秒で自動フェードアウト）

### UI オーバーレイの合成方法
FFmpeg のフィルタ機能で、ゲーム画面上にアバター + 配信 UI を合成する。

- 生存時間、最高記録、現在の目標、危険度、実況テキスト、**アバター表情状態**は、オーケストレーターが JSON ファイルに書き出す。
- 軽量な描画プロセス（Node.js + node-canvas）が JSON を読み取り、透過 PNG のオーバーレイ画像を定期更新する（200ms 間隔。アバターの口パクに対応するため）。
- FFmpeg の `overlay` フィルタで、ゲーム映像 + オーバーレイ PNG を合成して配信する。

### 音声パイプライン

```
[実況音声]                          [ゲーム音]
オーケストレーター                  Minecraft Client
  ↓ 実況テキスト                      ↓ 音声出力
VOICEVOX Engine                     PulseAudio sink_game
  ↓ WAV 音声データ                    ↓ (音量 30〜40%)
PulseAudio sink_voice               │
  ↓ (音量 100%)                      │
  └──────────┬───────────────────────┘
             ↓
FFmpeg (-f pulse -i sink_mix)
  → ゲーム映像 + アバター + UI オーバーレイと合成して配信
```

- **Minecraft のゲーム音は必須**。採掘音、足音、モブの鳴き声、環境音があることで臨場感が大幅に向上し、実況音声だけでは得られない「ゲームをプレイしている感」を生む。
- Minecraft Client の音声出力を PulseAudio 経由で FFmpeg にミックスする。
- 音量バランス: ゲーム音は実況音声の 30〜40% 程度の音量に設定し、実況の聞き取りを妨げないようにする。
- PulseAudio の仮想シンクを 2 つ用意: `sink_game`（Minecraft 音声）と `sink_voice`（VOICEVOX 音声）。FFmpeg で両方を入力し、音量を調整してミックスする。

### Bot 制御

| 項目 | 選定 | 理由 |
|---|---|---|
| ライブラリ | Mineflayer (Node.js) | Minecraft bot の事実上の標準。活発なメンテナンス |
| パス探索 | mineflayer-pathfinder | A* ベースの 3D パス探索 |
| 戦闘 | mineflayer-pvp | 攻撃・防御の基本制御 |
| 視界 | prismarine-viewer (デバッグ用) | 開発時のbot視点確認。本番では不要 |
| ワールドデータ | prismarine-world | ブロック・エンティティ情報の取得 |

Mineflayer は Minecraft サーバーにクライアントとして接続する。描画用の Minecraft クライアントとは別プロセスで動作する。つまり Minecraft サーバーには 2 つのクライアントが接続する構成になる:
1. **Mineflayer bot**: AI の判断に基づいて行動する。画面は持たない。
2. **Minecraft Client**: bot と同じ座標を追従し、映像描画のみを担当する。いわゆる「カメラマン」。

### 2 層行動判断アーキテクチャ

Bot の行動判断は 2 つの層に分離する。これにより、LLM の応答待ち中でも bot が安全に動き続けることを保証する。

#### Layer 1: リアクティブ層（即時応答、LLM 不使用）

Mineflayer にハードコードする反射的な行動ルール。ゲームティック単位（50ms）で評価し、ミリ秒で応答する。LLM は経由しない。

| トリガー | 行動 | 優先度 |
|---|---|---|
| クリーパー接近（5 ブロック以内） | 即座に後退・距離を取る | 最高 |
| HP が 6 (♥♥♥) 以下 | 食料を食べる。なければ安全な場所へ退避 | 最高 |
| 落下検知（足元に空洞） | 移動停止・後退 | 最高 |
| 溶岩・火災検知 | 回避方向へ移動 | 最高 |
| 攻撃を受けた | 攻撃元に応じて反撃 or 逃走（装備と敵の種類で判定） | 高 |
| 夜が来た（かつ拠点座標既知） | 拠点に帰還 / ベッドで寝る | 高 |
| 水中で酸素低下 | 水面に浮上 | 高 |
| インベントリ満杯 | 低価値アイテムを破棄 | 中 |

リアクティブ層のルールは生存に直結するものだけに絞る。判断ロジックはシンプルな条件分岐で実装し、AI の創造性が入る余地はない。

#### Layer 2: 戦略層（LLM、状況適応型の可変間隔）

「次に何をするか」「何を喋るか」の創造的・戦略的判断を LLM が担う。

| 判断内容 | 例 |
|---|---|
| 中目標の設定・更新 | 「鉄装備が揃った。次は拠点の防御を強化しよう」 |
| サブタスク選択 | 「木材が少ない。補充してから洞窟探検に向かおう」 |
| リスク判断 | 「装備が整っていないから洞窟は後回し」 |
| 実況テキスト生成 | 「鉄の剣が完成した。これで少しは戦えるかな」 |

#### 呼び出し間隔（状況適応型）

固定間隔ではなく、状況に応じて間隔を変える。

| 状況 | 間隔 | 理由 |
|---|---|---|
| 安定作業中（採掘、建築、農業） | 20〜30 秒 | 急な判断変更が不要 |
| 移動中・探索中 | 10〜15 秒 | 新しい情報が入りやすい |
| 戦闘直後 | 即時（イベント駆動） | 状況が大きく変化した |
| 新エリア到達 | 即時（イベント駆動） | 周囲の評価が必要 |
| 中目標達成時 | 即時（イベント駆動） | 次の方針を決める必要がある |

#### 2 層の連携フロー

```
時間軸 →

[LLM] "鉄鉱石を掘りに洞窟へ行こう"
  ↓ 行動プラン: 洞窟へ移動 → 鉄鉱石を採掘 → 帰還
  ↓
[Mineflayer] pathfinder で洞窟へ移動中...
  ↓
[リアクティブ] ゾンビ検知! → 即座に戦闘 → 撃破
  ↓ ※ 戦闘終了 = イベント → LLM 即時呼び出し
  ↓
[LLM] "ゾンビは倒したけど HP が減った。一度帰って回復しよう"
  ↓ 行動プラン変更: 帰還 → 回復 → 再度洞窟へ
  ↓
[Mineflayer] 拠点へ帰還中...
  ↓ (20秒後、定期呼び出し)
  ↓
[LLM] "食料も減ってきたし、先に小麦を収穫しておくか"
```

- bot は常に動いている。LLM の応答を待って停止することはない。
- LLM は「監督」、Mineflayer は「選手」。選手は反射的な判断を自分で行い、監督に定期的に方針を仰ぐ。
- リアクティブ層は LLM の行動プランを**上書きできる**（生存が最優先のため）。
- リアクティブ層が発動した場合、発動理由とともに LLM へのイベント通知をキューに入れ、次の LLM 呼び出しで状況を伝える。

### LLM

| 項目 | 選定 | 理由 |
|---|---|---|
| プロバイダ | Anthropic (Claude) または OpenAI (GPT-4o) | API ベースで GPU 不要。自前推論は複雑すぎる |
| 呼び出し頻度 | 戦略判断: 15〜30 秒ごと、緊急判断: 即時 | コストと応答性のバランス |
| 入力 | ゲーム状態 JSON（インベントリ、周囲のブロック・エンティティ、HP、位置、時刻、ペーシング情報） |
| 出力 | 次の行動指示 + 実況テキスト（1 回の応答で両方を返す） |

#### LLM コスト試算
- 1 回の呼び出し: 入力 ~2,000 トークン、出力 ~200 トークン
- 15 秒間隔: 1 時間に 240 回、1 日 5,760 回
- Claude Sonnet 相当: 入力 $3/MT, 出力 $15/MT
- 日次コスト: 入力 $0.035 + 出力 $0.017 = 約 $0.05/日
- **月次: 約 $1.5**（想定より大幅に安い。頻度を上げたり、より大きなコンテキストを渡しても十分収まる）

※ 実際にはシステムプロンプトや過去の行動履歴でトークン数が増える可能性があるため、月 $10〜30 程度を見込んでおく。

### TTS

| 項目 | 選定 | 理由 |
|---|---|---|
| エンジン | VOICEVOX Engine | 無料・オープンソース。日本語ネイティブ。CPU で動作 |
| 実行方式 | Docker (`voicevox/voicevox_engine:cpu-latest`) | 環境分離。インストール容易 |
| API | HTTP REST (localhost:50021) | オーケストレーターから HTTP で呼び出し |
| ボイス | 運用開始時に選定（ずんだもん、四国めたん等の中からキャラに合うものを選ぶ） |

VOICEVOX は CPU の約 50% を使用する設計。g4dn.xlarge の 4 vCPU 環境では約 2 コア分を消費する。音声合成のレイテンシは短文で 0.5〜1 秒程度。

### オーケストレーター

| 項目 | 選定 | 理由 |
|---|---|---|
| 言語 | Node.js (TypeScript) | Mineflayer と同一ランタイム。型安全性を確保 |
| 状態管理 | XState v5 または自作の軽量ステートマシン | セクション 13 の状態遷移を実装 |
| プロセス管理 | systemd | 各コンポーネントを systemd ユニットとして管理。Restart=always |
| 永続化 | SQLite (ローカル) | 世代番号・最高記録・配信 ID 等。軽量で十分 |
| YouTube 制御 | googleapis (Node.js) | 配信枠の作成・更新・終了 |

### プロセス構成の全体像

```
systemd
 ├── minecraft-server.service      # Minecraft Java Server
 ├── minecraft-client.service      # Minecraft Client (Xvfb + VirtualGL)
 ├── voicevox.service              # VOICEVOX Engine (Docker)
 ├── orchestrator.service          # オーケストレーター (本体)
 │    ├── Mineflayer Bot           # (子プロセスとして起動)
 │    ├── LLM API 呼び出し
 │    ├── TTS 呼び出し
 │    ├── YouTube API 呼び出し
 │    └── 状態管理・ヘルスチェック
 └── ffmpeg-stream.service         # FFmpeg (映像キャプチャ + 配信 + 録画)
```

- オーケストレーターが中心。Bot、LLM、TTS、YouTube の連携はすべてオーケストレーターが制御する。
- Minecraft Server/Client、VOICEVOX、FFmpeg は独立した systemd サービスとし、個別に再起動可能にする。
- オーケストレーターは他サービスのヘルスチェックを行い、応答がない場合は systemctl restart を発行する。

# 22. LLM 入出力フォーマット設計

### 設計方針
- LLM への入力・出力はすべて構造化 JSON とする。自然文の解析に頼らず、確実にパース可能な形式で通信する。
- LLM は 1 回の呼び出しで「次の行動プラン」と「実況テキスト」の両方を返す。2 回に分けない。
- 行動プランは高レベルな目標指示（「鉄鉱石を採掘する」）で記述し、Mineflayer の低レベル API への変換はオーケストレーター側で行う。LLM に Mineflayer の関数名を知らせる必要はない。

### システムプロンプト（常時固定部分）

システムプロンプトには以下を含める。これは毎回の呼び出しで変化しない。

- キャラクター定義（星守レイの人格、話し方、価値観）
- 行動原則（セクション 7 のルール）
- ペーシングの価値観（セクション 8 の判断基準）
- 利用可能な行動カテゴリの一覧と説明
- 出力 JSON フォーマットの仕様
- 「実況テキストはテンプレートではなく、今の状況に即した自然な思考を出力すること」という指示

### 記憶と学習の仕組み

死亡時に以下のプロセスで「教訓」を生成・蓄積する。

1. **死亡時**: オーケストレーターが死亡直前の状況（位置、HP 推移、周囲のエンティティ、直前の行動ログ 10 件）を収集する
2. **教訓生成**: LLM に「この死亡状況を分析し、1 文で教訓を要約せよ」と依頼する（通常のゲームプレイ用とは別の 1 回限りの呼び出し）
3. **永続化**: SQLite の `death_history` テーブルに世代番号・生存時間・死因・教訓を保存する
4. **次回以降の入力**: 毎ターンの LLM 入力に `memory.recent_deaths`（直近 5 件）を含める。古い履歴は要約して圧縮してもよい

この仕組みにより、星守レイは「前にクリーパーで死んだから夜は拠点に戻ろう」といった判断ができるようになる。人格はリセットされず、経験を重ねるキャラクターとして成長する。

### 出力フォーマット（LLM の応答）

```json
{
  "action": {
    "goal": "拠点に戻って就寝する",
    "reason": "夜になりゾンビが近くにいる。無理に採掘を続けるより安全を確保する",
    "steps": [
      "拠点へ帰還する",
      "ベッドで寝る",
      "朝になったら採掘を再開する"
    ]
  },
  "commentary": "...ゾンビの気配がする。夜の洞窟は危険だし、一度拠点に戻ろう。鉄はあと少しだけど、死んだら全部終わりだから。",
  "current_goal_update": null,
  "threat_level": "medium"
}
```

#### 各フィールドの役割
- **action.goal**: 今回の行動の目標（1 文で要約）
- **action.reason**: なぜその行動を選んだか（デバッグ・ログ用。配信には出さない）
- **action.steps**: 目標を達成するためのステップ（自然言語。順序付きリスト）
- **commentary**: 実況テキスト。VOICEVOX で音声化される。1〜3 文程度
- **current_goal_update**: 中目標に変更がある場合のみ記載（例: "ネザーを目指す"）。変更なしは null
- **threat_level**: "low" / "medium" / "high" / "critical"。配信 UI の危険度表示に使用

### action.steps の解釈

オーケストレーターは steps の自然言語を、事前定義されたアクションにマッピングする。

| 自然言語の例 | マッピング先 |
|---|---|
| 「拠点へ帰還する」 | pathfinder で base.position へ移動 |
| 「鉄鉱石を採掘する」 | 最寄りの iron_ore を検索 → 移動 → 採掘 |
| 「ベッドで寝る」 | ベッドブロックを右クリック |
| 「木を伐採する」 | 最寄りの oak_log を検索 → 移動 → 採掘 |
| 「食料を食べる」 | インベントリから食料を選択 → 食べる |
| 「かまどで精錬する」 | かまどブロック → 素材投入 → 完了待ち |
| 「周辺を探索する」 | ランダムまたは未踏方向へ pathfinder 移動 |
| 「洞窟に入る」 | 最寄りの洞窟入口を検索 → 進入 |

マッピングできない steps が返された場合は無視し、次の step に進む。解釈不能なプラン全体が返された場合は、前回のプランを継続する。

### エラーハンドリング
- LLM の応答が JSON としてパースできない場合: 前回のプランを継続。実況テキストは生成しない
- LLM API がタイムアウトした場合（5 秒以上）: 前回のプランを継続。次のサイクルでリトライ
- LLM API が連続で失敗した場合: リアクティブ層のみで動作を継続。実況は停止。復旧後に再開

# 23. AI 思考→行動→実況の 1 サイクル詳細フロー

### サイクル概要

1 サイクルは「ゲーム状態の取得」から「行動実行と実況音声の再生」までの一連の流れである。行動実行と音声再生は並列で行う。

```
[1] ゲーム状態取得
 ↓
[2] LLM プロンプト構築
 ↓
[3] LLM API 呼び出し (非同期)
 ↓
[4] レスポンス解析
 ↓ 分岐
 ├──[5a] 行動プラン → Mineflayer に指示 → 実行開始
 │
 └──[5b] 実況テキスト → VOICEVOX → 音声再生
       ↓
      [5c] UI 状態更新 (目標・危険度・実況テキスト)
 ↓
[6] 次のトリガーまで待機
 ↓ (定期 or イベント駆動)
[1] に戻る
```

### 各ステップの詳細

#### [1] ゲーム状態取得
Mineflayer の API から以下を収集する。所要時間: 数 ms。

- `bot.health`, `bot.food` → player.hp, player.hunger
- `bot.entity.position` → player.position
- `bot.inventory.items()` → inventory_summary
- `bot.nearestEntity()` → nearby_entities
- `bot.blockAtCursor()`, `bot.findBlocks()` → nearby_blocks_of_interest
- `bot.time.timeOfDay` → time_of_day
- `bot.isRaining` → weather
- ペーシング用タイマーから → pacing 情報
- イベントキューから → recent_events

#### [2] LLM プロンプト構築
[1] の情報をセクション 22 の入力フォーマットに整形する。

- インベントリは全アイテムを列挙せず、主要アイテムの要約にまとめる（トークン節約）
- nearby_entities は距離 20 ブロック以内に限定
- recent_events は直近 5 件に限定
- previous_plan は前回の LLM レスポンスから引き継ぐ

#### [3] LLM API 呼び出し
- 非同期で実行。呼び出し中も Mineflayer は前回のプランを実行し続ける
- タイムアウト: 5 秒。超過した場合はスキップし、前回プランを継続
- ストリーミングレスポンスは使用しない（JSON パースの都合上、完全なレスポンスを待つ）

#### [4] レスポンス解析
- JSON パース → バリデーション（必須フィールドの存在確認）
- パース失敗時は前回プランを維持（エラーハンドリング参照）

#### [5a] 行動プラン実行
- action.steps を順番にオーケストレーターが解釈し、Mineflayer の API 呼び出しに変換
- 各 step は完了するか、次の LLM サイクルが来るか、リアクティブ層が割り込むまで実行
- step 実行中にリアクティブ層が発動した場合、step は中断される。中断理由は次サイクルの recent_events に記録

#### [5b] 実況音声生成（[5a] と並列）
1. commentary テキストを VOICEVOX の `/audio_query` に送信（音声合成パラメータ取得）
2. `/synthesis` で WAV 音声データを取得
3. PulseAudio 仮想シンクに音声を再生
4. FFmpeg が PulseAudio から音声を取り込み、配信に合成

- 音声生成のレイテンシ: 0.5〜1.5 秒（テキスト長による）
- commentary が空文字列の場合は音声を生成しない（無言期間）
- 前の音声がまだ再生中の場合、キューに入れて順次再生する（被せない）

#### [5c] UI 状態更新（[5a] と並列）
- action.goal → UI の「現在の目標」表示を更新
- threat_level → UI の「危険度」表示を更新
- commentary → UI の「実況テキスト」表示を更新（字幕として数秒表示）
- current_goal_update → 中目標の表示を更新（変更がある場合のみ）
- これらは JSON ファイルに書き出し、オーバーレイ描画プロセスが読み取る

#### [6] 次のトリガーまで待機
次の LLM 呼び出しは以下のいずれかで発火する（セクション 21 の可変間隔に従う）。

- **定期タイマー**: 状況に応じた間隔（安定時 20〜30 秒、探索時 10〜15 秒）
- **イベント駆動**: リアクティブ層の発動、中目標達成、新エリア到達、戦闘終了など
- **ステップ完了**: action.steps の全 step が完了した場合（次の指示が必要）

いずれか最初に発火した条件でサイクルが再開する。

### 並列処理のタイミング図

```
時間 →
─────────────────────────────────────────────────────────

Mineflayer  [== 前回プラン実行中 ==][====== 新プラン実行 ======]
                                    ↑
LLM         .......[呼出][==応答待ち==][解析]
                                         ↓
VOICEVOX    .........................[音声生成][再生中....]
                                              ↓
FFmpeg      [===== 映像キャプチャ + 配信 (常時) ===== 音声合成 ====]

UI更新      ..............................[JSON書出][描画反映]
```

- Mineflayer は LLM を待たずに常時動作する
- LLM の応答が返ってきた時点で、行動プランの切り替えと音声生成が同時に開始する
- FFmpeg は常時動作しており、映像と音声をリアルタイムで合成・配信している

# 24. YouTube メタデータ仕様

### 配信タイトル
テンプレートで自動生成する。世代番号と試行の位置づけが伝わるシンプルな形式。

```
【AI Minecraft】星守レイのハードコア生存実験 #Gen{世代番号}
```

例:
- `【AI Minecraft】星守レイのハードコア生存実験 #Gen1`
- `【AI Minecraft】星守レイのハードコア生存実験 #Gen13`

最高記録を更新中の場合（前回最高記録の 80% を超えた時点）:
```
【AI Minecraft】星守レイのハードコア生存実験 #Gen{世代番号}【記録更新中】
```

タイトルの動的更新は YouTube Live Streaming API の `liveBroadcasts.update` で配信中に実行可能。

### 概要欄
テンプレートで自動生成する。ダッシュボードから編集可能。

```
🤖 AI VTuber 星守レイが Minecraft ハードコアモードに挑戦中。
死んだら終わり。どこまで生き残れるか、観測してください。

📊 現在の記録
- 今回: Gen{世代番号} (配信開始時に更新)
- 最高記録: {最高記録} 分 (Gen{最高記録世代番号})
- 通算試行回数: {世代番号 - 1} 回

🔗 チャンネル概要
AI がマインクラフトをプレイし、その思考と判断をリアルタイムで実況する実験的チャンネルです。

#Minecraft #AIVTuber #星守レイ #ハードコア
```

### サムネイル

テンプレートベースで自動生成する。

- **サイズ**: 1280 x 720
- **背景**: Minecraft のタイトル画面のスクリーンショット（事前に撮影し固定画像として保持）

# 25. 配信開始フロー詳細（BOOTING → PREPARING_STREAM → LIVE_RUNNING）

オペレーターがトリガーを発行（MANUAL モード）または AUTO モードの自動遷移で、以下の順序でプロセスが起動する。

### BOOTING フェーズ（プロセス起動）

```
1. オーケストレーター起動（または既に稼働中）
   ├── SQLite から前回状態を読み込み
   ├── 世代番号をインクリメント
   └── 状態を BOOTING に更新
        ↓
2. Minecraft Server 起動確認
   ├── 既に稼働中 → スキップ
   └── 未起動 → systemctl start minecraft-server
   └── localhost:25565 への接続が成功するまで待機（タイムアウト 60 秒）
        ↓
3. 新しいハードコアワールドを作成
   ├── server.properties で level-name を gen_{世代番号} に設定
   ├── hardcore=true, difficulty=hard を確認
   └── Minecraft Server をリロード or 再起動
        ↓
4. Minecraft Client 起動確認
   ├── Xvfb (:99, 1920x1080) が稼働中か確認
   ├── VirtualGL 経由で Minecraft Client を起動
   └── サーバーに接続し、描画が開始されるまで待機
        ↓
5. VOICEVOX 起動確認
   ├── localhost:50021 への HTTP ヘルスチェック
   └── 未応答 → docker start voicevox
        ↓
6. FFmpeg 起動
   ├── x11grab で :99 をキャプチャ開始
   ├── ローカル録画を開始（まだ YouTube には送出しない）
   └── BOOT / WAITING シーンのオーバーレイを表示
```

### PREPARING_STREAM フェーズ（配信枠準備）

```
7. YouTube 配信枠を作成
   ├── YouTube Live Streaming API: liveBroadcasts.insert
   ├── タイトル、概要欄、タグをテンプレートから生成（セクション 24）
   ├── サムネイルを生成しアップロード
   ├── プライバシー: public
   └── 配信 ID / ストリームキーを取得
        ↓
8. FFmpeg の RTMP 送出先を設定
   ├── YouTube のストリームキーを環境変数 or 設定ファイルで渡す
   └── FFmpeg を再起動（録画 + RTMP 同時出力モード）
        ↓
9. Mineflayer Bot をサーバーに接続
   ├── localhost:25565 に接続
   ├── スポーン完了を待機
   └── 初期状態を取得（位置、インベントリ、時刻）
        ↓
10. 初回 LLM 呼び出し
    ├── 「新しい世界に降り立った」状態のゲーム情報を送信
    ├── memory に過去の死亡履歴を含める
    └── 最初の行動プランと実況テキストを取得
```

### LIVE_RUNNING 移行

```
11. オーバーレイを LIVE シーンに切り替え
12. YouTube 配信のステータスを live に更新（liveBroadcasts.transition）
13. 状態を LIVE_RUNNING に更新
14. 生存タイマーを開始
15. ヘルスチェックループを開始
16. 1 サイクルループ（セクション 23）を開始
```

各ステップでエラーが発生した場合は RECOVERING 状態に遷移し、リトライを試みる。3 回連続失敗で IDLE に戻り、ダッシュボードにエラーを表示する。

# 26. 運用ダッシュボード

# 27. オーケストレーターの耐障害設計

### オーケストレーター自身のクラッシュ復旧

オーケストレーターは systemd の `Restart=always` で管理されるため、クラッシュ後は自動的に再起動される。再起動時の復旧フローは以下の通り。

```
1. オーケストレーター再起動
   ↓
2. SQLite から最後の状態を読み込み
   ├── last_state: LIVE_RUNNING
   ├── current_gen: 13
   ├── stream_id: xxxx
   ├── survival_start_time: 2026-03-28T14:00:00Z
   └── youtube_stream_key: yyyy
   ↓
3. 状態に応じた復旧分岐
   ├── IDLE → そのまま IDLE で待機
   ├── LIVE_RUNNING → 復旧フロー（下記）
   ├── BOOTING / PREPARING_STREAM → 最初からやり直し
   ├── DEATH_DETECTED / ENDING_STREAM → 終了処理を完遂
   └── COOL_DOWN → タイマー再計算して待機継続
   ↓
4. LIVE_RUNNING からの復旧:
   a. Minecraft Server の生存確認
   b. Mineflayer Bot の再接続（同じワールド）
   c. FFmpeg の再起動（同じストリームキーで再接続）
   d. YouTube 配信の継続確認
   e. 生存タイマーを survival_start_time から再計算
   f. LLM サイクルを再開
```

重要: **ワールドデータは Minecraft Server が保持している**ため、オーケストレーターがクラッシュしてもゲームの進行状態は失われない。Mineflayer が再接続すれば、同じワールドの同じ位置からプレイを継続できる。

### SQLite に書き込む状態更新のタイミング

| イベント | 書き込む内容 |
|---|---|
| 状態遷移時 | 新しい状態、タイムスタンプ |
| 配信開始時 | 世代番号、stream_id、stream_key、survival_start_time |
| LLM サイクルごと | 最新の action.goal、threat_level（上書き） |
| 死亡時 | 死因、生存時間、教訓 |
| ヘルスチェック異常時 | 異常内容、復旧アクション |

書き込みは WAL モード（Write-Ahead Logging）で行い、クラッシュ時のデータ破損を防止する。

# 28. 障害シナリオと復旧手順

| # | 障害 | 検知方法 | 自動復旧 | 配信への影響 |
|---|---|---|---|---|
| 1 | **Minecraft Server クラッシュ** | TCP 接続失敗 | systemd restart → Bot 再接続 → ワールド継続 | 数秒の映像フリーズ。ワールドは保持 |
| 2 | **Mineflayer Bot フリーズ** | 60 秒間行動ログなし | Bot プロセス再起動 → サーバー再接続 | bot が数秒停止。映像は表示され続ける |
| 3 | **FFmpeg クラッシュ** | RTMP バイト数停止 | systemd restart → 同じストリームキーで再接続 | 配信が数秒途切れる。YouTube 側は自動再接続 |
| 4 | **VOICEVOX ダウン** | HTTP ヘルスチェック失敗 | Docker restart | 実況音声が一時停止。映像・ゲームプレイは継続 |
| 5 | **LLM API 障害（数分）** | 5 回連続失敗 | リアクティブ層のみで行動継続。3 分後リトライ | 実況テキストが停止。bot はリアクティブ層で安全行動 |
| 6 | **LLM API 障害（長期）** | 10 分以上応答なし | リアクティブ層で生存維持。ダッシュボードに警告 | 実況なしで安全行動のみ。配信は継続するが面白みは低下 |
| 7 | **YouTube RTMP 切断** | ストリームステータス inactive | 同じキーで再接続。失敗時は新枠作成 | 数秒〜数十秒の配信途切れ |
| 8 | **YouTube API 障害** | API 呼び出しエラー | エクスポネンシャルバックオフでリトライ | 新規配信枠が作れない。録画は継続 |
| 9 | **EC2 インスタンス障害** | EC2 Auto Recovery | インスタンス自動再起動 → オーケストレーター復旧フロー | 数分の完全停止。EBS データは保持、NVMe は消失 |
| 10 | **ディスク容量不足 (NVMe)** | ヘルスチェックで空き容量監視 | 古い録画ファイルの強制削除 → S3 転送の即時実行 | 録画が停止する可能性。配信自体は継続 |
| 11 | **オーケストレーター自身のクラッシュ** | systemd が検知 | 自動再起動 → SQLite から状態復元（セクション 27） | 数秒の実況停止。ゲーム・映像は Minecraft/FFmpeg が独立稼働で継続 |

### 復旧できないケース（手動介入が必要）

- EBS の破損（SQLite DB の喪失）→ バックアップからの復元が必要
- AWS アカウントレベルの障害 → 待つしかない
- YouTube チャンネルの BAN / 制限 → 運用ポリシーの見直し
- Minecraft のアップデートによる Mineflayer の互換性破壊 → バージョン固定で回避

# 29. Minecraft バージョン方針
- 使用エディション: Java Edition
- バージョン: 実装開始時点で Mineflayer が安定サポートする最新版を採用する（2026 年 3 月時点では 1.21.x 系が想定）
- バージョンを固定し、Mineflayer の互換性が確認できるまでアップデートしない
- server.properties の `online-mode=false` で Mojang 認証をスキップ（ローカル専用のため）
